# WebRTC 原理学习项目包装与补齐计划

## 1. 项目定位

当前项目更适合包装成一个面向音视频技术支持岗位的“WebRTC 原理学习与协议分析实验项目”，而不是普通的音视频 Demo，也不应包装成真实线上环境里的通用解密排障平台。

建议项目名称：

- WebRTC 原理学习与协议分析实验项目
- WebRTC 协议链路学习与抓包分析项目
- WebRTC 互通验证与原理分析实验

更推荐第一种。它既能体现你对协议和链路的理解，又能避免把实验环境里的解密能力误写成真实工作中的通用能力。

---

## 2. 项目简介

该项目是一个用于系统学习 WebRTC 原理的实验项目。围绕 Native 端与 Web 端的互通实验，搭建了从信令、传输到媒体层的端到端分析链路。项目包含 Mac Native Demo、Web 信令与互通 Demo、Wireshark/tshark 抓包分析、在可控实验环境中导出 DTLS/SRTP 会话密钥、SCTP/DataChannel 明文验证、音视频流离线分析，以及相关分析文档沉淀。

相关解密与导出能力仅用于本地实验和协议学习，不对应真实生产环境中的通用解密能力。

如果用于简历或面试，重点不应放在“做了一个页面”或“做了一个 Demo”，而应放在以下三类能力：

- 主动学习和验证 WebRTC 原理的能力
- 协议、抓包、编解码、跨端联调的综合理解能力
- Python 工具整理与知识沉淀能力

## 3. 当前项目可以怎么拆给面试官看

### 3.1 跨端互通验证层

这部分对应岗位中的：

- 技术问题复现与定位能力
- 跨团队沟通时对链路边界的理解能力

### 3.2 抓包与协议分析层

作用：体现你对 WebRTC 关键机制的拆解和理解能力。

可以讲的内容：

- 基于 Wireshark 和 tshark 分析 WebRTC 全链路
- 关注 Signaling、ICE/STUN、DTLS、SCTP/DataChannel、SRTP、RTP 媒体流
- 在可控实验环境下通过 `pcap + keylog` 方式验证协议链路
- 能根据抓包判断问题是出在信令、方向映射、编解码还是封装层

这部分对应岗位中的：

- 熟悉 WebRTC 等流媒体协议
- 具备通过实验验证理解协议机制的能力

### 3.3 工具整理层

作用：这是把学习和分析过程沉淀成可复用材料的部分。

可以讲的内容：

- 编写 Python 脚本辅助 WebRTC 实验样本的离线分析
- 支持 DataChannel 明文验证、媒体流提取、日志汇总
- 将手工 Wireshark 点选过程整理成可复用命令行工具
- 形成 `输入抓包 -> 输出日志/样本分析结果` 的自动化流程

这部分对应岗位中的：

- Python 工具整理能力
- 日志分析与实验复盘能力
- 将学习案例抽象成可复用工具的能力

### 3.4 知识沉淀层

作用：对应岗位里“知识库、培训、赋能”的要求，也体现你的学习沉淀能力。

可以讲的内容：

- 沉淀了抓包分析文档、协议说明、问题复盘材料
- 形成从现象、原因、解决方案到复现方法的完整记录
- 具备把复杂技术问题整理成可传播材料的能力

这部分对应岗位中的：

- 编写知识库文档
- 支持团队培训和一线赋能

## 4. 和目标岗位的映射关系

### 4.1 已经比较匹配的部分

#### 主动学习与原理理解

你这个项目的核心价值不是“写了多少业务代码”，而是可以证明：

- 你会主动搭实验环境来理解复杂协议
- 你能面对复杂音视频问题时拆解链路
- 你知道该抓什么、看什么、怎么缩小范围
- 你能从现象追到机制，而不是停留在“只是会调用接口”

#### 工具和知识库沉淀

项目中已经有工具脚本和分析文档，这一点非常适合岗位要求中的：

- 主动识别和抽象学习中的共性问题
- 沉淀工具
- 编写知识库文档

#### WebRTC 协议与 Python

项目已覆盖：

- WebRTC 链路理解
- Python 自动化脚本开发
- Wireshark/tshark 使用
- 日志和抓包分析

### 4.2 当前还偏弱的部分

#### CDN / HTTP / DNS / HLS / RTMP

岗位不只看 WebRTC。当前你的材料更偏实时音视频链路，对以下方向表达不足：

- CDN 调度与缓存
- HTTP 与 DNS 问题排查
- HLS / RTMP / HTTP-FLV 等非 WebRTC 流媒体协议

#### Linux 在线问题定位

你现在的项目更偏本地实验和离线分析，还不够像“线上支持场景”：

- Linux 系统工具使用需要更体系化表达
- 线上日志、服务端链路、弱网/高延时/丢包等场景表达还不够

#### 编解码基础表达还不够显式

虽然你已经接触了 H264、VP8、RTP 封装、RED 等内容，但简历如果不主动写出来，面试官不一定会自动推断你有较扎实的编解码基础。

## 6. 如何把 FFmpeg / 编解码学习经历并入这个项目

你提到本地项目已经删除，但之前学习过这个 FFmpeg Android 项目：

- [FFmpegAndroid](https://github.com/xufuji456/FFmpegAndroid/tree/master/app/src/main)

从该仓库公开内容看，它至少覆盖以下学习方向：

- FFmpeg 命令行使用
- FFmpeg 编译与 NDK/JNI 集成
- FFmpeg 源码学习
- 多媒体基础知识
- 音视频协议基础
- 播放器相关框架认知

更稳妥的写法不是说“独立完成了一个播放器项目”，而是把它写成你在当前 WebRTC 学习项目之外，系统补充过的音视频基础能力，并把它并回到本项目的“技术准备”和“问题分析能力”中。

### 6.1 建议写法

可以在项目描述后补一条“相关能力补充”：

- 结合 FFmpeg Android 学习项目，系统梳理过 FFmpeg 命令行、Android NDK/JNI 集成、多媒体协议与编解码基础，对 H264、AAC、RTP 封装、转码/抽流/封装转换等音视频处理流程有实践性理解。

也可以写到“技能”部分：

- 熟悉 FFmpeg 常用命令与音视频处理流程，理解封装/转码/抽流、H264/VP8/AAC/Opus 等基础编解码概念。

### 6.2 更适合面试的表达方式

如果面试官问“你是怎么学 FFmpeg 的”，可以这样讲：

1. 为了补齐 WebRTC 学习项目中的媒体层认知，不只是停留在抓包和协议层，我系统学习过一个 Android 方向的 FFmpeg 学习项目。
2. 重点学习了 FFmpeg 命令行、NDK/JNI 集成、多媒体协议和编解码基础，帮助我更好理解 WebRTC 中提取出来的视频流为什么会解码失败、封装为什么不匹配、RTP 负载和实际编码格式为什么会出现偏差。
3. 这部分学习没有作为当前仓库保留，但它直接帮助我在当前实验项目中识别出“RED 外层 + VP8 主编码，而不是裸 H264”的问题。

这个说法的优点是：

- 不夸大
- 和当前项目强关联
- 能把学习经历转化成岗位需要的能力

### 6.3 可以写到当前项目亮点中的版本

你甚至可以把它并入项目亮点里：

- 在 WebRTC 抓包分析之外，结合 FFmpeg 与音视频编解码学习，能够从媒体封装、RTP 负载格式、码流导出与转封装角度辅助理解音视频链路。

## 7. 面试时怎么讲这个项目

### 7.1 30 秒版本

我做了一个 WebRTC 原理学习项目，重点不是业务功能，而是围绕 Native 和 Web 端互通，建立实验环境并把从信令、ICE、DTLS、DataChannel 到 SRTP/媒体流的关键链路走通和理解清楚。我还结合 FFmpeg 和编解码学习，把媒体层的异常也纳入分析范围，比如识别 RTP 负载和实际编码格式不一致的问题。

### 7.2 2 分钟版本

项目最初是为了系统理解 Native 和 Web 互通中的 WebRTC 关键机制。为了能稳定观察和验证协议链路，我先搭了一个双端互通实验环境，然后基于 Wireshark、tshark 和 Python 做离线分析工具，把学习链路打通到 Signaling、ICE/STUN、DTLS、SCTP/DataChannel、SRTP 和音视频导出。过程中我在可控实验环境里导出 DTLS 和 SRTP key，用来辅助理解加密传输后的协议层次和媒体数据流。后面结合 FFmpeg 和编解码基础学习，把媒体层也补齐了，所以可以进一步分析提取失败、码流封装异常、RTP 负载格式不匹配等问题。最终这个项目沉淀成了 Demo、脚本、抓包样本和分析文档的组合。

### 7.3 5 分钟版本

建议按下面的结构讲：

1. 目标：为什么要做一个 WebRTC 学习实验项目
2. 场景：Native 和 Web 的互通实验涉及哪些关键链路
3. 方法：如何抓包、如何在可控环境中导出 key、如何自动化分析
4. 典型案例：为什么最开始 H264 提取失败，最后发现其实是 RED + VP8
5. 沉淀：脚本、文档、方法论
6. 反思：补齐了哪些协议、编解码和工具链能力
