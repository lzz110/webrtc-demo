# 字节音视频技术支持专家面试手册设计

## 目标

基于当前 WebRTC 学习实验项目，输出一份偏理解型的音视频技术支持面试手册。材料需要帮助候选人准确描述项目、理解核心原理、应对逐层追问，并能把实验经验迁移到真实生产环境的排障思路中。

## 受众与使用方式

- 受众：准备面试字节跳动音视频技术支持专家岗位的项目作者。
- 使用方式：先学习项目主线，再按技术专题查漏补缺，最后通过模拟追问练习口头表达。
- 答案深度：核心题提供完整回答、实验依据、可能追问和简短回答。

## 内容结构

### 1. 项目表达

- 一句话项目定位。
- 简历三条描述。
- 一分钟和三分钟讲稿。
- 项目能力与岗位职责映射。

### 2. 项目证据索引

将面试表述映射到仓库中的实际材料：

- Native 与 Web 信令、Offer/Answer、ICE、DTLS：`docs/native2web-pcap-analysis.md`。
- Web 与 Web 信令和候选类型：`docs/web2web-pcap-analysis.md`。
- SDP 字段和协商：`docs/sdp_information_analysis.md`。
- DTLS/SRTP、DataChannel、媒体导出工具：`src/`。
- RED + VP8 与 H264 识别案例：`share.md` 和 `src/red_vp8_extract.py`。

### 3. 深度问题库

按 WebRTC 连接建立的真实顺序组织：

1. Signaling 与 Offer/Answer。
2. SDP、BUNDLE、MID、Payload Type 和 SSRC。
3. ICE、STUN、TURN、NAT 与候选选择。
4. DTLS 角色、证书指纹和密钥协商。
5. SRTP、RTP、RTCP、丢包与抖动。
6. SCTP/DataChannel 可靠性与 PPID。
7. H264、VP8、Opus、RED、FFmpeg 与封装格式。
8. Python、Wireshark、tshark 工具化。

### 4. 技术支持场景题

覆盖无法建联、无声音、单向音频、卡顿、首帧慢、弱网、DNS/HTTP/CDN、HLS/RTMP、Linux 日志和网络排查等场景。每个场景使用统一结构：

- 先确认影响范围和复现条件。
- 按信令、网络、传输、媒体、播放分层定位。
- 明确需要的日志、指标、抓包和环境信息。
- 给出临时止损、根因定位和长期改进思路。

### 5. 模拟面试

以“面试官提问 → 候选人回答 → 深挖追问 → 回答要点”的方式，串联项目主线和岗位扩展能力。

### 6. 边界与风险

- 明确项目是个人学习实验，不包装成生产项目。
- 明确抓包解密依赖本地可控环境和主动导出的 key，不声称能够解密真实客户网络包。
- FFmpeg Android 经历表述为系统学习和实践，不表述为独立开发完整播放器。
- 对未在项目中验证的 CDN、HLS、RTMP、Linux 线上经验，使用“理解原理/掌握排查方法”，不虚构生产经验。

### 7. 复习计划

提供一周优先级安排，优先确保项目主线、WebRTC 建联、典型案例和生产排障方法可以口述，再补 CDN、流媒体协议和 Linux 工具。

## 质量标准

- 每个项目结论能指向仓库中的实验材料或明确标注为理论扩展。
- 不包含无法自证的生产经历或量化成果。
- 核心问题不仅给定义，还说明协议之间的因果关系和排障顺序。
- 内容可直接用于简历修改、口头复习和模拟面试。

## 验证方式

- 检查文档中的项目事实是否与现有抓包分析和脚本一致。
- 扫描是否存在“可解密真实线上流量”“生产平台”等越界表述。
- 检查核心问题是否同时包含完整回答、实验依据、追问和简短回答。
- 检查 Markdown 标题、链接和代码引用是否有效。
